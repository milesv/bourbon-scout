# Bourbon Scout

Allocated bourbon inventory scraper with Discord webhook alerts. Monitors 11 retailers for 46 bottles (45 rare/allocated + 1 canary health check) near a zip code, sends per-store Discord alerts with @here pings when bottles are found, plus a quiet summary with per-scraper health metrics after each scan.

## Architecture

- **`scraper.js`** — Main entry point. Contains all scrapers, Discord alerting, and poll orchestration. Schedule-aware poll loop: 30 min default (4 AM – 10 PM MT), 20 min boost (Tue/Thu nights), sleeps 10 PM – 4 AM. ±3 min jitter. Single-run via `RUN_ONCE=true` for CI. Scans all stores concurrently with a limit of 8.
- **`com.bourbon-scout.plist`** — macOS launchd daemon config. Keeps scraper running with auto-restart. Logs to `~/Library/Logs/bourbon-scout.log`.
- **`scripts/install-daemon.sh`** / **`scripts/uninstall-daemon.sh`** — Install/remove the launchd daemon.
- **`browser-profiles/`** — Per-retailer persistent Chrome profile directories (auto-created). Contains HTTP cache, service workers, IndexedDB, visited history. Gitignored.
- **`lib/bottles.js`** — Phase 1 of modularization. Pure data + pure functions: `SEARCH_QUERIES`, `TARGET_BOTTLES`, `CANARY_NAMES`, `CANARY_BY_RETAILER`, `isCanaryFor`, `BOTTLE_INTEREST_TIERS`, `ULTRA_RARE_BOTTLES`, `IGNORED_BOTTLES`, `bottleInterestTier`. Designed to be importable from any future per-retailer module without circular-import risk (no env-var reads, no network, no module state). `scraper.js` imports and re-exports these for backward compatibility with tests.
- **`lib/blocked.js`** — Phase 2 of modularization. Bot-detection / block-page heuristics + PerimeterX challenge solver: `isBlockedPage`, `isFetchBlocked`, `isCostcoBlocked` (alias), `FETCH_BLOCKED_PATTERNS`, `solveHumanChallenge`. Inlines its own jittered `sleep` to stay dependency-free.
- **`lib/match.js`** — Phase 2 of modularization. Bottle matching + price/size parsing + Discord title truncation: `normalizeText`, `EXCLUDE_TERMS`, `matchesBottle`, `parsePrice`, `dedupFound`, `parseSize`, `truncateTitle`, `DISCORD_TITLE_LIMIT`, `filterMiniatures`, `MIN_BOTTLE_PRICE`, `MAX_BOTTLE_PRICE`. Pure functions / pure constants.
- **`lib/http.js`** — Phase 2 of modularization. HTTP fingerprint helpers: `IS_MAC`, `CHROME_VERSION` (auto-detected from system Chrome), `CHROME_PATH`, `getGreasedBrand`, `FETCH_HEADERS`, `fetchRetry`. `scraperFetch` / `scraperFetchRetry` stay in `scraper.js` because they're tightly coupled to the proxy-exhaustion state machine — future per-retailer modules import `FETCH_HEADERS` from here and call `scraperFetch` from scraper.js.
- **`lib/geo.js`** — Zip-to-coordinates (`zipToCoords`) via zippopotam.us API and `haversine` distance calculation. No dependencies.
- **`lib/discover-stores.js`** — Auto-discovers nearby stores for all retailers given a zip code and radius. Caches results to `stores.json` with a 7-day TTL. Uses vanilla `playwright-core` with `--disable-blink-features=AutomationControlled` for browser-based store locators and Kroger REST API for Kroger.
- **`lib/fallback-stores.js`** — Static hardcoded store data for the 85283 area. `FALLBACK_STORES` used when browser-based store locators fail (e.g., CI datacenter IPs get blocked). `EXTRA_STORES` contains user-specified stores outside the discovery radius (e.g., Costco Scottsdale #427, Costco Paradise Valley #1058) — always merged after discovery, deduped by storeId.
- **`state.json`** — Per-store stock state, nested as `{ "retailer": { "storeId": { "bottles": { "name": { url, price, sku, firstSeen, lastSeen, scanCount } }, "lastScanned" } } }`. Auto-generated, safe to delete for a fresh start.
- **`stores.json`** — Cached store discovery results. Auto-generated, safe to delete to force re-discovery.
- **`metrics.jsonl`** — Append-only JSONL scan metrics log. One JSON line per scan with per-retailer health (queries/ok/blocked/failed), canary hits, bottles found, scan duration. Used by `loadRecentMetrics()` to compute 24h trends shown in Discord summary embed. `pruneMetrics(30)` runs at startup to remove entries older than 30 days. Auto-generated, safe to delete (loses history only).
- **`.github/workflows/bourbon-scout.yml`** — GitHub Actions cron job (currently disabled in favor of local daemon). Schedule: 30 min default, 20 min on Tue/Thu nights (Wed/Fri UTC), active 5 PM – 10 AM MST only. Self-hosted Mac runner (`bourbon-scout-mac`). Caches state files between runs. Uses `RUN_ONCE=true`. 25-minute timeout.
- **`debug/`** — Screenshots, HTML dumps, and CSS selector lists captured from retailer store locator pages. Used for fixing selectors when sites change. Generated by `debug-locators.js`.

## Key Patterns

- **ESM throughout** — `"type": "module"` in package.json, all files use `import`/`export`
- **Clean browser mode (all scrapers)** — All 5 browser scrapers use `launchRetailerBrowser(key, { clean: true })` which launches Chrome using vanilla `playwright-core` — truly unpatched CDP with zero rebrowser modifications. `rebrowser-playwright-core`'s CDP patches (`__re__getMainWorld`, `__re__emitExecutionContext`) caused intermittent "Cannot find context" and "session closed" protocol errors that corrupted page evaluations mid-flight. **Key discovery**: stealth+rebrowser CDP modifications are fingerprinted by both PerimeterX and Akamai as automation signals — removing them makes the browser undetectable. Defaults to headed on Mac (`headless: !IS_MAC`). On Mac, launches system Chrome (`/Applications/Google Chrome.app`) via `executablePath` for authentic TLS fingerprint. Falls back to bundled Chromium on other platforms. Launches with `--disable-blink-features=AutomationControlled` and `ignoreDefaultArgs: ['--enable-automation']`. Browser contexts get platform-aware Chrome UA + `Sec-CH-UA` Client Hints matching `FETCH_HEADERS`. **Minimized on Mac** — CDP `Browser.setWindowBounds` minimizes the window after launch so it doesn't steal focus during scans.
- **Vanilla playwright everywhere** — Both retailer scrapers (`{ clean: true }`) and `discover-stores.js` use vanilla `playwright-core` with `--disable-blink-features=AutomationControlled`. `rebrowser-playwright-core` + `playwright-extra` + `puppeteer-extra-plugin-stealth` are still installed (used by the shared `newPage()` browser) but NOT used for any retailer scraper or store discovery.
- **PerimeterX challenge solver** — `solveHumanChallenge(page)` detects and solves PerimeterX "Press & Hold" challenges. Checks `document.body.innerText` for "Press & Hold" text, waits for `#px-captcha` button to render (1-2s), moves mouse naturally to center, holds for 14-18s (base, with ±30% jitter = 10-23s actual, guaranteeing ≥10s minimum), adds micro-movements (±2px every ~2s) during hold for realistic telemetry, releases, waits for navigation. Retries once on failure with fresh position. Called after homepage pre-warm and on blocked search pages for Total Wine, Costco, Walgreens, and Walmart. Only works when captcha.js renders the interactive button (requires clean browser mode).
- **Per-retailer browser mutex** — `acquireRetailerLock(key)` serializes browser access per retailer via promise chains. Prevents multiple stores from opening concurrent pages on the same persistent context, which anti-bot systems flag as automated behavior. Different retailers can run concurrently.
- **Browser fail-fast** — `retailerBrowserBlocked[key]` flag set when first store's browser attempt fails. Remaining stores skip browser entirely since same IP/profile means they'll fail too. Reset each poll.
- **Auto-detected Chrome version** — `CHROME_VERSION` reads the system Chrome binary's major version at module load via `execFileSync`. `FETCH_HEADERS` UA, `Sec-CH-UA`, and `Sec-CH-UA-Platform` all use the detected version. Prevents TLS-vs-UA version mismatch when Chrome auto-updates (e.g., system Chrome is 146 but headers claim 145). Falls back to a hardcoded version if Chrome isn't found.
- **Platform-aware User-Agent** — `IS_MAC` detection at module level. macOS Chrome UA + `Sec-CH-UA-Platform: "macOS"` on Mac, Windows Chrome on Linux/CI. Prevents UA-vs-TLS fingerprint mismatch that bot detectors flag.
- **Lazy browser launch** — Browser is only launched when a scraper actually needs it (via `newPage()`). When all scrapers succeed via fetch-first, no browser is started.
- **Atomic state writes** — `saveState()`, `saveBrowserState()`, `saveCache()` (discover-stores) all write to `.tmp` then `rename()` (atomic on POSIX). Prevents corruption on crash mid-write.
- **Browser cookie persistence** — `loadBrowserState()`/`saveBrowserState()` persist Playwright `storageState` to `browser-state.json`. In-memory cache (`browserStateCache`) avoids re-reading from disk on every `newPage()`. Cookies accumulate across polls, reducing bot detection signals.
- **Parallel query execution** — API scrapers (Kroger, Safeway) run all 14 queries via `Promise.all` with 50ms staggered starts. Fetch scrapers (Costco, TotalWine, Walmart) use batched parallelism via `runWithConcurrency(queryTasks, 2)`. Sam's Club uses `runWithConcurrency(productTasks, 2)`.
- **Kroger pagination** — Fetches page 2 (`filter.start=50`) when first page returns exactly 50 results, catching allocated bottles that spill past the first page. Same pattern used by Safeway. Page 2 failures are isolated in their own try/catch — a page 2 JSON parse error logs a warning but doesn't lose page 1 data or mark the query as failed.
- **Kroger fulfillment.inStore as in-stock signal (post-2026 API change)** — Kroger removed `inventory.stockLevel` from spirits responses in 2026. The old `stockLevel != null` guard caused 100% false negatives across all Fry's stores (canary detection went to 0%). Replaced with `fulfillment.inStore === true` as the primary signal. Still respects `inventory.stockLevel` when present (rejects "OUT_OF_STOCK" variants case-insensitive) but treats null/missing inventory as a non-blocker. Recovered Kroger canary from 0% → 100%, surfaced finds for EHT, Blanton's, Stagg, Elmer T. Lee, even Pappy 15/23 at Marketplace stores.
- **Kroger product URL slug** — `buildKrogerProductUrl(product, productId)` uses the API's `productPageURI` field for slug-based URLs (`/p/george-t-stagg-15-year-bourbon/0008800402784`) instead of bare-productId URLs (`/p/0008800402784` 404s). Strips the `?cid=...` analytics param. Falls back to the productId-only URL when `productPageURI` is missing. Run as a one-time backfill on `state.json` to repair already-recorded URLs from the bare-id era.
- **Kroger B+C tier classification (lead vs confirmed)** — `fulfillment.inStore: true` correlates with **planogram slot allocation**, not real shelf stock. To reduce false positives: `classifyKrogerFind(product, item)` returns `{ confidence: "confirmed" | "lead", aisle, facings, dataAgeDays }`. **Confirmed** = `numberOfFacings >= 3` OR `effectiveDate` within 14 days (well-stocked OR recent activity). **Lead** = single allocated slot with stale price data (most common at Marketplace stores carrying allocated bottles whose shelf is currently empty). Confirmed finds → green @here urgent alert; leads → yellow quiet alert with "call ahead" guidance. Eliminates the Higley/Stagg-style false positive where a single planogram slot fires alerts for an empty shelf.
- **Two-tier Discord embeds** — `buildStoreEmbeds()` splits `changes.newFinds` by `confidence` field. Confirmed finds get the green "🚨 NEW FIND" embed (urgent, with `_urgent: true` sidecar). Leads get a yellow "🔍 LEAD" embed (quiet, "store has a planogram slot but inventory signal is weak. Call ahead before driving."). Caller in `recordResult` filters embeds by `_urgent` to dispatch via `sendUrgentAlert` (with @here) vs `sendDiscordAlert` (silent). Non-Kroger retailers default to `confidence === "confirmed"` so existing behavior is preserved.
- **Multi-store consolidation** — When the same bottle is found at multiple stores during a single scan, the FIRST store fires immediately as urgent @here. Subsequent stores get buffered into `secondaryFindsBuffer` (Map keyed by `${retailer}:${bottle}`). At end of poll, one consolidated "🎯 ALSO AT — Pappy 15 (3 stores)" embed lists all secondary store addresses + Maps links. Reduces Discord spam during boost windows when allocations hit several Marketplace Fry's simultaneously without sacrificing first-detection latency.
- **Price-change tier alert** — `computeChanges()` emits `priceChanges[]` for bottles already in stock with ≥10% price movement (uses parsed numeric prices from `priceHistory`). New purple "💰 PRICE CHANGE" embed shows drops (~~$80~~ → **$60**, -25%) with strikethrough formatting plus rises. Drops surface clearance events; rises sometimes signal fresh-inventory spike pricing. Fires regardless of new finds — independent signal layer.
- **Daemon liveness heartbeat** — Optional `HEALTHCHECK_URL` env var. Daemon fires-and-forgets a GET to that URL at end of every successful `poll()`. External service (e.g., healthchecks.io free tier) emails/notifies when the ping doesn't arrive within configured grace period. Catches daemon crashes within ~90 min vs. waiting for manual log inspection.
- **Bandwidth tracking** — `trackBandwidth(retailerKey, byteSize)` accumulates fetch response bytes per retailer in a module-level `bandwidthStats` object. `retailerKeyForUrl(url)` heuristically attributes by hostname. Wired into `scraperFetch` to count every fetch-path response. Helps spot DataImpulse billing drift before 407 TRAFFIC_EXHAUSTED hits. Exposed via `_getBandwidthStats()` for observability scripts.
- **Log rotation at startup** — `maybeRotateLog()` runs at daemon main(). If `~/Library/Logs/bourbon-scout.log` exceeds 50MB, renames to `.log.old` (one backup retained) and starts fresh. macOS launchd writes append-only without rotation; without this the log grew unbounded (observed 11MB after a few weeks; would hit GBs over a year).
- **Reddit intel post scoring** — Heuristic score (retailer-keyword +2, bottle-keyword +2, recency<1h +2, upvotes>=1 +1, comments>=1 +1, ISO/question pattern −3). Posts below threshold 4 get logged + skipped instead of alerting. Cuts false-positive pings from people *asking* about bottles or referencing old drops, without LLM cost.
- **Two-tier retailer cooldown** — Acute (3 consecutive failures → 30-min cooldown) and chronic (12 consecutive failures → 6-hour cooldown). Chronic tier saves proxy bandwidth on long-broken retailers (WAF migration that takes days to resolve). Both auto-recover on first success.
- **Rich Kroger context in alerts** — `formatBottleLine` renders `🗺️ {aisle}` (e.g., LIQUOR 119, Shelf 5), `🔢 {N} facings`, and `⚠️ Inventory data is N days old — call to confirm` when `dataAgeDays > 60`. Helps user triage at a glance.
- **Marketplace-first scan order for Kroger** — `prioritizeStores("kroger", stores)` puts Fry's Marketplace locations ahead of regular Fry's Food And Drug in the task queue. Marketplace stores have larger spirits sections and receive most allocated drops. Stable sort preserves discovery order within each tier. Only applied to Kroger; other retailers unchanged.
- **Kroger HTML verify (disabled)** — `verifyKrogerCandidatesViaWebsite()` was an experiment to scrape `frysfood.com` product pages and parse the "Item Unavailable" widget for stronger inventory signal. Wired into `scrapeKrogerStore` behind `KROGER_HTML_VERIFY` env var (default `false`). Akamai blocks every modality-preferences POST (429 cpr_chlge → 403), so the verification never succeeds. Graceful fallback (returning candidates unchanged) means it's safe but useless. Kept the code but disabled by default. Re-enable only if Akamai bypass solved.
- **EXCLUDE_TERMS false positive prevention** — `matchesBottle()` rejects product titles containing sampler, gift set, variety pack, combo pack, bundle, miniature, or mini bottle before checking `searchTerms`. Prevents products like "Pappy Van Winkle Sampler Set" from triggering 4+ bottle matches at once.
- **Miniature bottle filter** — `filterMiniatures()` applied in `recordResult()` removes bottles priced under `MIN_BOTTLE_PRICE` ($20) or with explicit small sizes (<200ml). Catches 50ml miniatures that pass EXCLUDE_TERMS because the title doesn't mention the size (e.g., Total Wine listing "Blanton's Original" at $12.49 with no "50ml" in the name). Bottles with no price are kept (can't confirm miniature).
- **Concurrent scanning** — All stores run concurrently via `runWithConcurrency(tasks, 8)`. Browser and API scrapers are unified in the same task queue.
- **Structured data extraction over CSS selectors** — Scrapers prefer embedded JSON (Walmart `__NEXT_DATA__`, Total Wine `window.INITIAL_STATE`) over fragile CSS selectors. Costco uses stable MUI `data-testid` attributes. CSS selector fallbacks exist but are last resort.
- **Residential proxy support** — Optional `PROXY_URL` env var routes fetch scraper traffic through a proxy. Supports HTTP/HTTPS and SOCKS5/SOCKS4 protocols — `createProxyAgent()` auto-detects based on URL scheme. Playwright browser proxy requires URL parsing into `{ server, username, password }` (can't pass the full URL). Discord webhooks skip the proxy. **On Mac, browser paths skip the proxy entirely** — residential IP + real Chrome is the strongest anti-detection setup. Proxy IPs are more likely to be flagged by PerimeterX/Akamai than residential ones. Browser proxy is only used on CI (datacenter IPs). With proxy set, Walmart fetch-first works on CI too.
- **Proxy auto-failover** — When primary proxy returns 407 (TRAFFIC_EXHAUSTED), `failoverToBackupProxy()` auto-switches ALL retailers to `BACKUP_PROXY_URL`. `isProxyAvailable()` returns true as long as at least one proxy provider is alive. Both `primaryProxyExhausted` and `backupProxyExhausted` flags must be set before fetch paths are disabled. Browser fallback (direct IP on Mac) takes over only when both are exhausted. Reset at the start of each `poll()`.
- **Sticky proxy sessions** — `refreshProxySession()` called at the start of each `poll()`. For DataImpulse, switches to a random sticky port (10000-20000) so all requests in a scan share one residential IP for ~30 min. Prevents IP rotation from breaking cookie chains (e.g., Costco homepage pre-warm cookies tied to one IP, search requests from another).
- **Mid-poll proxy rotation** — `rotateRetailerProxy(key)` switches a retailer to a fresh sticky port (= new residential IP) when the current IP is detected as burned. Triggered automatically after 2 consecutive fetch failures in Costco, Walmart, and Sam's Club fetch paths. Clears stale cookies from the old IP's session — both `browserStateCache` (force reload from disk) and the active browser context's cookies (`context.clearCookies()`). Anti-bot ties cookies to IPs, so sending burned-IP cookies from a new IP is a detection signal. Proxy URL is read dynamically per-request (`getRetailerProxyUrl()`) so rotation takes effect immediately for remaining queries.
- **Per-retailer IP isolation** — Each retailer gets its own sticky proxy port (= its own residential IP) via `retailerProxyAgents`. The general `proxyAgent` uses a separate port (`basePort + retailers.length`) to avoid sharing with any retailer. Prevents cross-retailer IP flagging cascades — if Costco's IP gets burned, Walmart's is unaffected. `getRetailerProxy(key)` returns the per-retailer fetch agent; `getRetailerBrowserProxy(key)` returns Playwright-compatible `{ server, username, password }`.
- **Persistent browser profiles** — `launchRetailerBrowser(retailerKey)` uses `chromium.launchPersistentContext(profileDir)` to maintain full Chrome profiles per retailer in `browser-profiles/{key}/`. HTTP cache, service workers, IndexedDB, and visited history persist on disk across polls — the scraper looks like a returning visitor. Within a poll, `retailerBrowserCache` shares the context across stores. Pages are closed after each use; contexts closed at end of poll via `closeRetailerBrowsers()`. Profile data survives context close for next poll. On launch, stale `SingletonLock`/`SingletonSocket`/`SingletonCookie` files are auto-cleaned to prevent hangs after Chrome crashes.
- **Browser launch failure isolation** — All 7 wrapper functions (`scrapeCostcoStore`, `scrapeTotalWineStore`, `scrapeWalmartStore`, `scrapeWalgreensStore`, `scrapeSamsClubStore`, `scrapeSafewayStore`, `scrapeAlbertsonsStore`) catch `launchRetailerBrowser` failures with try/catch. On failure: logs error, tracks health as "fail", returns `[]`. Prevents a single Chrome crash from taking down the entire poll via `runWithConcurrency`.
- **Albertsons multi-store mutex** — `scrapeAlbertsonsStore` wraps its work in `acquireRetailerLock("albertsons")` to serialize concurrent Albertsons stores through the shared persistent browser context. Without the mutex, when Store A's retry logic called `context.close()`, Stores B and C immediately hit "page.evaluate: Target page, context or browser has been closed" because they were holding pages from that context. Same pattern as Total Wine and Walmart. Single-store retailers (Safeway) don't need this.
- **Browser profile recovery** — `launchRetailerBrowser` catches `launchPersistentContext` failures (e.g., "Cannot find context", CDP protocol errors from corrupted profiles). On first failure, nukes the profile directory (`rm -rf browser-profiles/{key}/`) and retries once with a fresh profile. Loses cached cookies/history but is better than a dead scraper. Uses `_retried` flag to prevent infinite loops.
- **Walgreens retry on block** — Since Walgreens is `scrapeOnce`, a single Akamai challenge wipes out all stores. `scrapeWalgreensStore` retries once with a fresh browser context if ≥50% of queries were blocked on the first attempt. Clears `retailerBrowserCache` and adds a 3-5s cooldown between attempts. Uses **consecutive** failure tracking (`wgConsecutiveBlocks >= 4`) instead of cumulative — a transient block that resolves doesn't count against the budget. Counter resets to 0 on each successful query.
- **Walgreens false-positive defense (5-layer)** — Walgreens AZ stores often don't carry liquor at all (different licensing per location), so search-result cards regularly leak catalog ghosts. Empirically, the daemon's entire history shows zero real allocated finds at Walgreens — only repeating "Weller Antique 107" hits. Five layers, deepest-first: **(L1) Tightened DOM selector** — `STRICT_SELECTOR = ".card__product, [class*='card__product']"` (dropped the `[data-testid*='product']` fallback, which matched related-product carousels). Divergence telemetry compares strict-vs-broad card counts each query and warns when broad >> strict (early signal of UI redesign). **(L2) Positive in-stock evidence** — flipped from "absent OOS text → assume in-stock" to "must contain pickup/cart cue" via `hasPositiveSignal` field on each card; carousel ghosts have no such cue and are skipped with logging. **(L3) Repeat-find suppression** — `applyWalgreensConfidenceDemotion` demotes any non-canary find to `lead` when 3+ events for that bottle exist in the last 30 days at Walgreens (`state._walgreensFpTracking.fpHistory[name]`). Real allocated drops don't recur; chronic catalog ghosts cycle through state.json. **(L4) Cross-bottle freshness** — same helper demotes when no non-canary find at Walgreens in 14+ days (`state._walgreensFpTracking.lastNonCanaryFind`). Treats "first non-canary find in 2 weeks" as suspicious. **(L5) PDP fallback verification** — `verifyWalgreensFindsViaPDP` navigates to each find's product detail page, parses body text for negative signals (`not sold in stores`, `online only`) and positive signals (`pickup available`, `available at this store`). Negative-only → suppress; positive → mark `_pdpVerified: confirmed`; ambiguous → demote to lead; page error → keep original (fail-open). Bounded by 60s outer timeout to prevent runaway latency. The tracking ledger is updated post-emission via `recordWalgreensFpEvent` so this scan's events influence next scan's confidence (avoids self-demotion). All five layers preserve the canary path — Buffalo Trace at Walgreens is never demoted/suppressed since it's the legitimate retailer-health probe.
- **Priority-based query rotation** — `PRIORITY_QUERIES` (van winkle, george t stagg, eagle rare 17, thomas handy sazerac, eh taylor, michters bourbon) run every scan for fastest detection of time-sensitive bottles. Remaining queries alternate even/odd via `scanCounter % 2`. Each scan fires ~11 queries (6 priority + ~4 rotating + canary). All 46 bottles covered within 2 consecutive scans; high-value bottles checked every 30 min instead of every 60 min.
- **Query order randomization** — `shuffle(getQueriesForScan(SEARCH_QUERIES))` via Fisher-Yates before every scraper loop to avoid predictable access patterns that anti-bot ML models flag.
- **Chrome TLS fingerprint impersonation (got-scraping)** — All scraper fetch paths use `got-scraping` from Apify instead of `node-fetch`. `got-scraping` impersonates Chrome's TLS handshake (JA3/JA4 fingerprint, cipher suites, extensions order), making HTTP requests indistinguishable from real Chrome at the network layer. `scraperFetch()` wraps `gotScraping()` with a node-fetch-compatible response interface (`{ ok, status, headers, text(), json() }`). `scraperFetchRetry()` adds one-retry-on-error. API scrapers (Kroger, Safeway) and Discord webhooks still use raw `node-fetch` (no bot detection). Proxy passed as `proxyUrl` string option (not `agent` object).
- **Fetch-first with browser fallback** — All 5 browser-based scrapers try `got-scraping` fetch first (with Chrome TLS fingerprint) before spinning up Playwright. Walmart parses `__NEXT_DATA__`, Sam's Club parses `__NEXT_DATA__` per-product, Costco parses `data-testid` tiles via cheerio, Total Wine extracts `INITIAL_STATE` JSON (requires cached browser `_px*` cookies — cold fetch is always blocked by PerimeterX), Safeway calls the product search API with cached `_abck` cookies. All fall back to clean browser instances if blocked. Fetch paths require ≥25% of queries to succeed (`validPages >= ceil(totalQueries/4)`) before trusting results — prevents a single valid page from suppressing browser fallback when most queries are blocked.
- **Per-product-URL scraping** — Sam's Club search excludes spirits, so `SAMSCLUB_PRODUCTS` maps bottle names to product IDs. Each product page is fetched individually at `/ip/{productId}` instead of using SEARCH_QUERIES. Only bottles with known Sam's Club pages are checked.
- **Fetch retry** — `fetchRetry()` (for `node-fetch` API paths) and `scraperFetchRetry()` (for `got-scraping` scraper paths) retry once on network errors with 1s backoff and `console.warn` on first failure.
- **Brace-counting JSON extraction** — Total Wine INITIAL_STATE and Walmart `__NEXT_DATA__` are both extracted from HTML using a character-by-character brace counter that tracks string context and escape characters, safely handling `</script>` inside JSON string values.
- **Scrape-once pattern** — Costco search has no store filter, so results are identical across warehouses. Walgreens uses `USER_LOC` cookie with zip coordinates — nearby stores share results. Sam's Club product pages show availability for the nearest club. All three scraped once and broadcast to all stores.
- **Broad search queries** — 16 queries cover all 44 allocated bottles + 1 canary query = 17 total. API scrapers (Kroger, Safeway) use these instead of per-bottle queries (~60% fewer API calls).
- **Per-retailer bottle filtering** — `matchesBottle(text, bottle, retailerKey)` checks optional `retailers` field on TARGET_BOTTLES entries. Bottles with `retailers: ["costco", "totalwine", "walmart"]` only match at those retailers. Bottles without the field match everywhere (backward-compatible). Used for Michter's 10 Year and Penelope bottles (Costco + Total Wine + Walmart only).
- **Total Wine shipping option blacklist** — `isTotalWineShippingOption(option)` rejects shopping options whose `name`/`type` contains "ship". Allocated bourbon at Total Wine cannot be shipped — but the API's `shoppingOptions: [{name: "Ship", eligible: true}]` was firing in-stock alerts even when the actual store had nothing. Fix: in `matchTotalWineInitialState`, the in-stock check accepts `stockLevel.stock > 0` OR a non-shipping eligible option (Pickup/In-store/Delivery). Shipping is also stripped from the displayed `fulfillment` field on alerts. Preserves "may be available in store" Friday-morning detection while killing the ship-only false positive.
- **Total Wine match diagnostic log** — When `matchTotalWineInitialState` matches a bottle, logs which signal fired: `via stockLevel:N` (per-store ground truth), `via option:Pickup` (store-local "may be available"), or `via transactional` (national fallback, weakest). Helps audit alert quality at a glance — `stockLevel` finds are drive-now confidence, `option:Pickup` finds suggest call-first.
- **Canary bottle (Buffalo Trace)** — Always-in-stock bottle used as a scraper health check. Found canary = scraper is working. Missing canary = scraper may be broken. Canary results are filtered out before alerts/state — only shown in the health section of the summary embed. Canary is also excluded from the OOS list in Discord embeds (`TARGET_BOTTLES` filtered by `CANARY_NAMES` in `buildStoreEmbeds`).
- **Canary-driven retry** — After the main scan completes, retailers that ran queries but missed the canary get retried once. Closes browser cache, rotates proxy IP, then re-scrapes. scrapeOnce retailers re-broadcast to all stores; per-store retailers retry first store only. Retries run via `runWithConcurrency(retryTasks, 4)`.
- **Browser → fetch cookie chain** — `cacheRetailerCookies(key)` extracts cookies (including Akamai `_abck` and PerimeterX `_px*`) from browser contexts via `storageState()` after successful browser scrapes. `getCachedCookies(key)` returns cached cookies within a 1-hour TTL. Fetch pre-warm paths merge cached browser cookies with homepage `Set-Cookie` headers, giving fetch requests browser-quality cookies without JS execution. Applies to Costco, Walmart, Sam's Club, Total Wine (enables fetch path that's otherwise always blocked by PerimeterX), and Safeway (enables direct API calls that Akamai otherwise blocks).
- **Per-scraper health tracking** — `scraperHealth` module-level object tracks queries succeeded/failed/blocked per retailer per poll. `trackHealth(retailerKey, outcome, { query, via, storeId })` called in every scraper's query loop (including early-exit paths like Walgreens coords failure). Extended optional params track per-query success rates (`queryStats`), fetch-vs-browser attribution (`path.fetch`/`path.browser`), and per-store health (`stores`). Reset each poll. Summary embed always shows all 7 retailers as inline Discord fields (3 per row): ✅/⚠️/❌ for retailers with data, ⏭️ for skipped/missing ones.
- **Persistent scan metrics** — `appendMetrics()` writes one JSON line per scan to `metrics.jsonl` with per-retailer health stats (queries/ok/blocked/failed), canary hits, bottles found, scan duration, plus extended fields: `path` (fetch/browser attribution counts) and `queryStats` (per-query success rates). `loadRecentMetrics(hours)` reads the last N hours of entries. `computeMetricsTrend(scans)` aggregates stats across scans. Discord summary embed shows a 24h trend line (per-retailer success %, canary %, bottles found) when ≥2 scans exist.
- **Peak find time analysis** — `computePeakHours(recentMetrics)` analyzes 14 days of metrics to identify which day-of-week + hour slots have the highest bottle find rates. Returns null until 100+ scans exist for statistical significance. Top 5 peak slots shown in Discord summary embed. Informs future dynamic boost period adjustment.
- **Shared Kroger OAuth token** — Fetched once per poll via singleton promise (`fetchRetry` for network resilience), shared across all Kroger stores. Clears cached token on 401 for automatic re-auth. Uses promo price when available (falls back to regular).
- **Per-scraper browser timeout with health tracking** — `withTimeout(promise, 300000, null)` wraps all browser scraper calls (Costco, Total Wine, Walmart, Walgreens, Sam's Club, Safeway, Albertsons). On timeout (5 min), wrappers log a warning, call `trackHealth(key, "fail")`, and return `[]`. **Bumped from 180s → 300s in 2026-04** after metrics showed 0% canary at Costco/Safeway/Sam's Club: pre-warm alone takes 130-180s on modern WAFs (Akamai/PerimeterX/Incapsula keep the page chatty so `networkidle` never settles), leaving zero budget for queries. 300s gives ~120-170s of query time after pre-warm completes.
- **Homepage pre-warm with humanization** — All browser scrapers (Costco, Total Wine, Walmart, Walgreens, Sam's Club, Safeway) visit the retailer homepage before searching. Uses `networkidle` wait + 3-5s randomized dwell time to let anti-bot sensor scripts (Akamai `_abck`, PerimeterX `_px*`) fully execute and phone home. After dwell, `humanizePage(page)` scrolls down via mouse wheel events, hovers over a random link via `page.locator("a[href]").nth(idx)` (uses `$$eval` to count links without materializing all DOM elements — `page.$$("a[href]")` hangs for 10-70s on link-heavy pages like Walgreens), then scrolls back up using wheel events (not `scrollTo` — that's a JS teleport detectable by anti-bot sensors). After humanization, `navigateCategory(page, retailerKey)` visits the spirits category page to build a natural browsing pattern (homepage → category → search). Costco, Walmart, and Sam's Club fetch paths also pre-warm, collecting `Set-Cookie` headers for subsequent requests.
- **Category page navigation** — `navigateCategory(page, retailerKey)` navigates to a retailer-specific spirits/liquor category page between homepage pre-warm and search queries. `CATEGORY_URLS` maps retailer keys to category pages (e.g., Costco `/search?dept=all&keyword=spirits`, Total Wine `/c/spirits/`, Safeway spirits aisle). Builds a realistic homepage → browse → search behavioral pattern that improves bot detection scores. Wired into all 6 browser scrapers after `humanizePage()`. 2-4s dwell time with `networkidle` wait.
- **Staggered retailer start times** — In `poll()`, each retailer's first task sleeps 10-30s (randomized) before starting, so all 7 retailers don't hit the internet simultaneously. First retailer starts immediately; subsequent ones stagger.
- **Direct product URL checks** — `loadKnownProducts(state)` merges seed URLs (`SEED_PRODUCT_URLS`) with previously found bottle URLs from `state.json` at the start of each poll. Three retailers have known URL checking: `checkWalmartKnownUrls(store)` fetches `__NEXT_DATA__` from product pages with `store_id` param, `checkCostcoKnownUrls()` does cookie-enhanced fetch (homepage pre-warm + cached Akamai `_abck` cookies) and parses via `matchCostcoTiles` or `matchCostcoProductPage` (og:title/price meta fallback for product detail pages), `checkTotalWineKnownUrls(store)` fetches product pages with `storeId` param and parses `INITIAL_STATE` JSON. All three run before search, merge results via `dedupFound`, and return known URL finds even if search/browser fails. Costco also has `checkCostcoKnownUrlsViaBrowser()` as a last-resort browser fallback when both fetch known URLs and browser search fail — navigates directly to product pages (less bot-targeted than search) with a 2-minute time budget and 3-blocked abort.
- **Seed product URLs** — `SEED_PRODUCT_URLS` provides 28 Costco + 29 Walmart + 40 Total Wine pre-discovered product page URLs for cold-start coverage. Merged into `knownProducts` during `loadKnownProducts()` so known URL checking works even with a fresh `state.json`. Costco URLs constructed from Warehouse Runner item numbers and previously known shelf tag numbers in `.product.{id}.html` format.
- **Schedule-aware scanning** — Active 4 AM – 10 PM MT, sleeps 10 PM – 4 AM. 30 min default intervals, 20 min "boost" on Tue/Thu nights into Wed/Fri mornings. `getMTTime()` uses `Intl.DateTimeFormat` with `America/Phoenix` timezone. `isActiveHour()`/`isBoostPeriod()` determine interval. ±3 min jitter per cycle. `parsePollIntervalMs()` still parses `POLL_INTERVAL` as fallback for `RUN_ONCE` mode.
- **Local daemon** — `com.bourbon-scout.plist` launchd config with `KeepAlive: true`. `scripts/install-daemon.sh` / `scripts/uninstall-daemon.sh` for setup. Logs to `~/Library/Logs/bourbon-scout.log`.
- **Adaptive retailer skipping** — `shouldSkipRetailer(key)` / `recordRetailerOutcome(key, success)` track consecutive failures per retailer. After 3 consecutive polls with <25% query success rate, the retailer is skipped for 30 min to let IP reputation recover. Cooldown auto-expires.
- **Human-like inter-query pacing** — Browser scrapers wait 2-4s (randomized) between search queries. All fetch scrapers (Costco, Walmart, Sam's Club) wait 1.5-3s between queries. Real humans take 5-15s between searches, so this interval is still faster but much less suspicious than sub-second bursts.
- **Fetch-only-ok health tracking** — Fetch paths (Costco, Walmart, Total Wine, Sam's Club) only track health on success (`trackHealth(key, "ok")`), not on failure. When fetch fails and browser fallback runs, only browser events appear in health metrics. Prevents double-counting inflation (e.g., "0/69" from fetch+browser both tracking failures for 5 Walmart stores).
- **Backup proxy provider** — `BACKUP_PROXY_URL` + `BACKUP_PROXY_RETAILERS` env vars let specific retailers use a different proxy provider (e.g., IPRoyal for Costco if DataImpulse IPs are burned at Akamai). Each backup retailer gets its own sticky port on the backup proxy. Unused retailers stay on the primary `PROXY_URL`.
- **Costco fetch resilience** — `scrapeCostcoViaFetch` has layered anti-bot countermeasures: (1) cookie pre-warm from homepage, (2) adaptive concurrency (1 for SOCKS5, 4 for HTTP), (3) inter-query jitter (250-500ms), (4) expanded blocked detection via `isCostcoBlocked()` with 8 Akamai patterns, (5) retry-with-backoff on blocked responses (2-3s pause before retry).
- **Bot detection** — `isBlockedPage()` checks page title AND body text for challenge/CAPTCHA/access-denied/security-check indicators. Browser scrapers skip queries that hit bot detection instead of recording false "empty" results. `isFetchBlocked()` (shared across all fetch paths) detects 12 Akamai/PerimeterX patterns including `_ct_challenge`, `px-captcha`, `Request unsuccessful`, `Incapsula`, `Enable JavaScript`, `verify you are human`, `security check`, `one more step`, `checking your browser`. Case-insensitive, scans first 10K chars for performance. `isCostcoBlocked` is a backward-compatible alias. Applied in Costco, Walmart, and Sam's Club fetch paths before JSON extraction.
- **Per-retailer Sec-Fetch-Site** — First query per scraper sends `cross-site` + Google Referer (simulating search click). Subsequent queries send `same-origin` + retailer domain Referer (simulating internal navigation). Uses `.map()` index (`i > 0`) instead of runtime flag to avoid race conditions in concurrent task execution.
- **Chrome-accurate FETCH_HEADERS** — Property insertion order matches Chrome's HTTP/2 wire order (Client Hints before User-Agent — Akamai pattern-matches header order as a zero-CPU bot signal). `Accept` header matches real Chrome (includes `image/avif`, `image/webp`, `application/signed-exchange`). `Sec-Fetch-User: ?1` and `priority: u=0, i` (HTTP/2 priority signal) included. `Accept-Encoding` is `gzip, deflate` only (Brotli removed for compatibility).
- **Dynamic Sec-CH-UA brand string** — `getGreasedBrand(majorVersion)` replicates Chromium's GREASE algorithm from `user_agent_utils.cc`. Each Chrome version deterministically selects a grease brand name, version, and entry ordering from three lookup tables. Auto-matches whenever Chrome auto-updates — prevents brand/version mismatch that bot detectors flag.
- **State pruning** — `pruneState()` removes stale retailer/store entries from `state.json` on each poll, keeping state clean when stores change. Top-level `_`-prefixed tracking keys (`_redditMentions`, `_redditSeen`, `_watchList`, `_walgreensFpTracking`) are explicitly preserved — without that guard they'd get deleted every poll and only survive because the corresponding feature re-populates them immediately, which silently erased history for Reddit-mention dedup, watchlist notification status, and Walgreens FP tracking. The `_`-prefix convention is now the canonical way to add cross-poll tracking data.
- **Text normalization** — `normalizeText()` converts Unicode curly quotes/apostrophes to ASCII before matching, preventing silent misses when retailers use `\u2019` in product names.
- **Fetch sanity checks** — Walmart fetch path validates `__NEXT_DATA__` contains a real `searchResult` structure before trusting results. Prevents challenge pages with valid JSON but no search data from being treated as "nothing in stock."
- **AbortSignal timeouts** — All fetch calls (Discord 10s, zipToCoords 10s, Kroger/Safeway/Walmart 15s, Kroger store locator 15s) use `AbortSignal.timeout()` to prevent hung requests from blocking concurrency slots.
- **Error isolation** — `runWithConcurrency` wraps tasks in `.catch()` so one failing task never aborts remaining tasks. Costco broadcast loop has per-store try/catch. `saveState`/`closeBrowser`/summary send are all individually error-handled.
- **Store discovery is separate from scraping** — `discover-stores.js` handles finding stores, `scraper.js` handles checking inventory
- **State change tracking** — `computeChanges()` diffs previous and current scan results. `updateStoreState()` persists `firstSeen`, `lastSeen`, `scanCount` per bottle per store. Re-alerts every N scans via `REALERT_EVERY_N_SCANS`.
- **Color-coded Discord embeds** — Green `@here` for new finds, orange for OOS losses, blue re-alerts, purple scan summary. Summary includes scanned store list (grouped by retailer) when nothing is found. 429/5xx/network retry with backoff (5 attempts, throws after exhaustion to prevent silent alert loss). Embed titles truncated at 256 chars, descriptions at 4096 chars.
- **Google Maps links** — Store addresses are hyperlinked to Google Maps universal search URLs.
- **Priority query retry** — After the main query loop in Costco and Walmart fetch scrapers, failed `PRIORITY_QUERIES` are retried once with a fresh delay. Sam's Club retries failed `PRIORITY_SAMSCLUB_PRODUCTS` (Pappy, BTAC). Only fires when other queries succeeded (validPages > 0) — avoids wasting time when the entire IP is burned.
- **Graceful shutdown** — SIGTERM/SIGINT handler waits up to 60s for the current poll to finish before exiting. Prevents lost Discord alerts on `launchctl stop`. `shuttingDown` flag + interval check on `polling` state.
- **Discord retry hardening** — `postDiscordWebhook` retries 5x on 429 rate limits, 5xx server errors, AND network errors (DNS, timeout, connection reset). A Pappy alert can't be lost to a transient Discord outage.
- **Startup env validation** — `validateEnv()` checks `process.env` at startup and warns about missing DISCORD_WEBHOOK_URL, ZIP_CODE, Kroger/Safeway credentials, and proxy on non-Mac.
- **Per-poll time budget** — 25-minute budget. The main scan is wrapped in `withTimeout(runWithConcurrency(tasks, 8), 25 * 60 * 1000, "POLL_TIMED_OUT")` — a hard kill that prevents browser-context cascades from producing 6+ hour scans (observed in May 2026 metrics: latest poll ran 21,295s before this fix). If the main scan exceeds 22 min (< 3 min remaining), canary retries are skipped to prevent cascading delays into the next poll. On hard timeout, partial data is preserved and a loud warning fires with granular reason counts so degradation alerts surface next scan.
- **Poll watchdog (auto-restart)** — Module-level `consecutivePollTimeouts` counts consecutive `POLL_TIMED_OUT` events; resets to 0 on the first non-timed-out poll. After 3 consecutive timeouts (`POLL_TIMEOUT_RESTART_THRESHOLD`), the daemon fires a Discord notification and calls `process.exit(1)` so launchd `KeepAlive=true` brings it back with a fresh event loop, fresh proxy state, no orphaned browser contexts. The 25-min single-poll timeout catches one hung scan; the watchdog catches a recurring systemic hang that the in-process timeout can't recover from. `_resetConsecutivePollTimeouts()` / `_setConsecutivePollTimeouts(n)` test helpers exist; the actual `process.exit` branch isn't driven in tests because mocking `process.exit` in vitest is risky.
- **Per-retailer phase timing** — `scraperHealth[key].phases` accumulates ms across three buckets: `prewarm` (homepage + category navigation + humanization), `queries` (the search loop), `extra` (known-URL fetches, PDP verification, browser fallbacks). `recordPhase(key, name, durationMs)` instruments the major browser-heavy scrapers (Walgreens, Costco, Walmart, Total Wine). Phases persist in `metrics.jsonl` and surface in the Discord summary embed when scan duration > 10 min OR any retailer's prewarm exceeded its query phase by > 30s — the canonical "pre-warm ate the budget" signal that diagnoses 25-min timeout caps. Sorts by total time descending and shows the top 5.
- **Discord fail-open buffer** — `postDiscordWebhook` already retries 5× on rate limits / server errors / network errors, but historically threw after exhaustion, losing the alert forever. Now wraps the inner `postDiscordWebhookOnce` and on terminal failure appends the payload to `pending-alerts.jsonl` (gitignored). At the start of every poll, `replayPendingAlerts()` reads the buffer, skips entries older than 24h, prefixes the first embed title with `⏳ [+Nm]` so the delay is visible, and posts up to `PENDING_REPLAY_LIMIT` (20) per cycle. Remaining entries persist for the next replay attempt. The asymmetry is brutal: cost of replaying a stale alert ≈ 0; cost of missing a real Pappy 23 alert = bottle gone. `postDiscordWebhookOnce` (the bufferless variant) is used during replay to avoid an infinite buffer-replay-buffer loop.
- **Bandwidth cost projection** — `formatBandwidthCost(bytes, sinceTs)` multiplies cumulative `bandwidthStats.totalBytes` by `PROXY_COST_PER_GB` (default $1.0, env-overridable). Projects daily / monthly burn from elapsed wall-clock since module load. Surfaces in the Discord summary embed as `💸 Bandwidth: X GB ($Y) │ proj. $Z/day │ $W/mo` when cost > $0.01. Today bandwidth is invisible until DataImpulse 407s; this makes it a leading indicator rather than a lagging post-mortem.
- **State.json corruption recovery** — atomic `.tmp` + `rename()` already prevents torn files at the OS level, but a process crash mid-`JSON.stringify` can still leave malformed content. `loadState` now tries the main file first; on parse failure, falls back to `state.json.bak`. `saveState` snapshots the previous good state to `.bak` *before* overwriting (validates parseability first — won't preserve corrupt data as a "known-good" fallback). One-generation backup is exactly what we want: recovers from "current is garbage but last save was fine" without unbounded versioning.
- **Browser context leak detector** — `closeRetailerBrowsers()` checks each persistent context's open page count before closing; warns and force-closes if `> 3` (empirical threshold; warmups may briefly hold 2 during a retry). Without this, a missed `page.close()` in a finally block accumulates orphaned pages, slowing Chrome to a crawl over a long-running daemon. Defensive `try`/`catch` guards `pages?.()` since not all context mocks implement the call.
- **Restock detection (`♻️ RESTOCK` tier)** — `state[retailer][store].oosHistory[bottleName] = { lastGone: ts }` records each transition to OOS. `computeChanges` checks the gap when emitting newFinds; gap ≥ 7 days promotes the find with `restock: true, lastGoneOOS: ts, restockGapDays: N`. `buildStoreEmbeds` renders restocks in a separate pink/magenta urgent (@here) embed — pre-empting the regular green NEW FIND tier so a restock isn't double-rendered. The 7-day threshold avoids triggering on Akamai-induced state flips where state.json briefly empties. `oosHistory` is capped at 100 entries per store via LRU eviction (defensive bound; TARGET_BOTTLES has 46 so the cap is purely against schema drift). Entries are *cleared* when the bottle re-appears, so an immediate next-scan disappearance doesn't re-trigger restock.
- **Phone numbers in lead/restock alerts** — `formatStoreInfo` returns `phoneLine` (a clickable `tel:` link) when `store.phone` is present on the store record. Rendered in lead embeds (where "call ahead" matters) and restock embeds (where confirming via phone before driving is high-value). Stripped to digits-only for the `tel:` URI; original formatting preserved for display. Stores can populate `phone` via `lib/fallback-stores.js` or future per-retailer discoverers — the rendering layer just shows it when available, no daemon-restart required.
- **`networkidle` → `domcontentloaded` everywhere** — All six browser scrapers (Costco, Total Wine, Walmart, Sam's Club, Walgreens, Albertsons; Safeway already migrated) now use `waitUntil: "domcontentloaded"` for homepage pre-warm. Modern WAFs (Akamai, PerimeterX, Incapsula) keep beaconing forever — chat widgets, telemetry, analytics — so `networkidle` never settles and eats the full 20s `goto` timeout. Explicit dwell `await sleep(5000-9000ms)` after the goto gives sensors the time they actually need. Net pre-warm savings: ~15-18s per scraper per scan; across 5 browser-heavy retailers × ~5 stores = ~5-8 min off the 25-min budget.
- **Page-closed bail-out in browser scrapers** — When `withTimeout` fires on a per-store scraper, the underlying loop kept calling `page.goto()` on a dead page, producing 30+ "Target page closed" errors per timeout. Each browser scraper (Sam's Club / Albertsons / Safeway) now checks `page.isClosed?.()` at the top of each iteration and breaks early. Defensive optional-chaining handles test mocks that don't implement `isClosed()`.
- **TotalWine `parseTotalWineInitialState` helper** — Extracts the inline JS object from product page HTML. Two real-world wrinkles handled: (1) `</script>` inside JSON string values (brace-counting state machine), (2) JS literal `undefined` embedded in object values (regex `:\\s*undefined\\s*([,}\\]])` → `:null$1` before `JSON.parse`). Total Wine started doing #2 in 2026, silently breaking every known-URL re-check that used `JSON.parse` directly. Returns null on any failure rather than throwing.
- **Watch list (human intelligence)** — `WATCH_LIST` array in scraper.js for rumored drops at specific stores. Each entry triggers a one-time @here Discord notification (gold embed) with store addresses and bottle list. `bottle` field is optional — omit for "Allocated Bourbon" which lists all tracked bottles at that retailer (respects per-retailer restrictions). Notification state persisted in `state.json._watchList`.
- **Reddit intel scraper** — Monitors r/ArizonaWhiskey and r/arizonabourbon via Reddit's public JSON API (no auth, no proxy needed). Polls every scan for posts less than 2 hours old matching target bottle names, store names, and drop keywords (kok, btac, pappy, allocated, drop, etc). Sends @here gold Discord embed with post title, text, author, score, and matched keywords. Deduplicates via `state._redditSeen` (pruned to 7 days). Facebook's Arizona Bourbon Society group is private — requires manual relay via WATCH_LIST.

## Environment Variables

All config is in `.env`:
- `ZIP_CODE` — center point for store discovery (currently 85283 / Tempe, AZ)
- `SEARCH_RADIUS_MILES` — how far to search (default 15)
- `MAX_STORES_PER_RETAILER` — caps per chain (default 5)
- `POLL_INTERVAL` — fallback scan interval as cron expression (default `*/30 * * * *`). Only used in `RUN_ONCE` mode; daemon mode uses schedule-aware logic (30 min default, 20 min boost).
- `REALERT_EVERY_N_SCANS` — re-alert interval for bottles still in stock (default 4)
- `KROGER_CLIENT_ID`, `KROGER_CLIENT_SECRET` — Kroger API OAuth credentials
- `KROGER_MAX_STORES` — Override `MAX_STORES_PER_RETAILER` for Kroger/Fry's only. Kroger is pure API (no browser, no anti-bot), so more stores cost only API calls. Default: inherits `MAX_STORES_PER_RETAILER`.
- `KROGER_RADIUS_MILES` — Override `SEARCH_RADIUS_MILES` for Kroger/Fry's only. Default: inherits `SEARCH_RADIUS_MILES`.
- `KROGER_HTML_VERIFY` — Enable Fry's product-page widget verification of Kroger candidates. Default `false` (Akamai blocks the modality preferences POST every time, so verification fails and just adds 30-60s/store). When `true`, after the public API surfaces a candidate, the daemon launches a browser, attempts to set the user's preferred store via `/atlas/v1/modality/preferences`, and parses the rendered "Item Availability" widget. Confirmed → green @here urgent alert. Refuted → silently filtered. Verification failure → graceful fallback to B+C tier classification.
- `SAFEWAY_API_KEY` — Safeway product search API key (also used as fallback for Albertsons when `ALBERTSONS_API_KEY` not set)
- `ALBERTSONS_API_KEY` — Optional Albertsons API key. Same Azure APIM key format as Safeway (same parent company). Falls back to `SAFEWAY_API_KEY` when not set.
- `DISCORD_WEBHOOK_URL` — Discord webhook for alerts
- `PROXY_URL` — Residential proxy URL. Supports HTTP (`http://user:pass@gw.dataimpulse.com:823`) and SOCKS5 (`socks5://user:pass@host:1080`). Routes browser and fetch scraper traffic through proxy. Discord webhooks are NOT proxied. DataImpulse sticky sessions use ports 10000-20000 (auto-configured by `refreshProxySession()`).
- `BACKUP_PROXY_URL` — Optional backup proxy URL (e.g., IPRoyal). Used by retailers listed in `BACKUP_PROXY_RETAILERS`.
- `BACKUP_PROXY_RETAILERS` — Comma-separated list of retailer keys to route through `BACKUP_PROXY_URL` instead of `PROXY_URL` (e.g., `costco,totalwine`).
- `SECONDARY_ZIPS` — Optional comma-separated list of additional zip codes for store discovery (e.g., `85054,85260`). Stores discovered in secondary zips are merged with primary results, deduped by storeId. Useful for covering adjacent metro areas without increasing `SEARCH_RADIUS_MILES`.
- `HEALTHCHECK_URL` — Optional liveness ping URL (e.g., healthchecks.io). Daemon GETs this URL after every successful poll; the external service emails/notifies if no ping arrives within configured grace period. Catches daemon crashes within ~90 min instead of leaving them invisible until manual log inspection. Free for 20 checks at healthchecks.io. No-op when unset.
- `PROXY_COST_PER_GB` — Override the default `$1.00 / GB` rate used in the Discord summary's bandwidth cost projection. Read at format time (no daemon restart required). DataImpulse default; set lower if you negotiate a different rate.
- `RUN_ONCE` — Set to `true` for single-run mode (used by GitHub Actions)

## Gotchas

- Costco's search URL (`/s?keyword=`) returns identical results regardless of warehouse — that's why we scrape once and broadcast.
- Total Wine uses PerimeterX (HUMAN Security) — cold fetch is always blocked (PerimeterX requires JS sensor execution). Cookie-enhanced fetch path re-enabled: when cached browser `_px*` cookies are available (from a prior browser scrape), `scrapeTotalWineViaFetch` merges them with homepage `Set-Cookie` headers to bypass PerimeterX. Falls back to clean browser when cookies are stale or unavailable.
- **stealth plugin + rebrowser-patches CAUSE detection** — PerimeterX and Akamai fingerprint the CDP modifications these tools make. Plain Chrome with `--disable-blink-features=AutomationControlled` passes undetected. All 5 browser scrapers use `clean: true` mode and `discover-stores.js` both launch via vanilla `playwright-core` (truly unpatched CDP). The stealth-wrapped `rebrowser-playwright-core` + `addExtra` chromium is only used for the shared `newPage()` browser.
- Walmart fetch-first gets blocked on CI (datacenter IPs) but works locally. With `PROXY_URL` set, fetch-first works on CI too. Without proxy, fetch path skips on CI and falls back to browser.
- Walmart fetch validates `__NEXT_DATA__` has a `searchResult` key — challenge pages with valid JSON but empty search data correctly fall back to browser instead of reporting "nothing found."
- Store locator CSS selectors in `discover-stores.js` break when retailer sites update their DOM. Kroger's API locator is the most stable.
- Safeway's store locator is at `local.safeway.com/search.html` — NOT `safeway.com/stores/search` (that 404s).
- Deleting `state.json` is safe but will send a full report for all stores on next poll.
- The Safeway scraper and Kroger scraper silently skip if their API keys aren't set.
- `stores.json` cache invalidates on zip/radius change or after 7 days.
- `lib/fallback-stores.js` provides static store data when browser-based locators fail on CI.
- Walgreens has no embedded JSON (`__NEXT_DATA__`, `INITIAL_STATE`) — server-rendered HTML only. Akamai blocks direct fetch (403), so it's browser-only with no fetch-first path. Store context set via `USER_LOC` cookie (base64 JSON: `{"la":"<lat>","lo":"<lng>","uz":"<zip>"}`).
- Sam's Club search (`samsclub.com/search?q=`) does NOT return spirits — product pages exist at `/ip/{slug}/{id}` but are excluded from search results. The scraper uses `SAMSCLUB_PRODUCTS` (a static map of bottle→productId) instead of `SEARCH_QUERIES`. Not all 38 bottles have Sam's Club pages — 14 are currently mapped (including Thomas H. Handy, Elmer T. Lee, Pappy 10/12/15/20/23 Year, and Buffalo Trace canary).
- NordVPN SOCKS5 proxy (`phoenix.us.socks.nordhold.net:1080`) doesn't work for scraping — it caps concurrent connections, causing `ConnectionRefused`/`ERR_SOCKS_CONNECTION_FAILED` under load.
- `PROXY_URL` supports both HTTP and SOCKS5 via `createProxyAgent()` auto-detection, but SOCKS5 proxies with connection limits (like NordVPN) will fail under concurrent load. Use a residential HTTP proxy (DataImpulse at $1/GB recommended).
- DataImpulse sticky sessions are **port-based** (10000-20000), NOT username-suffix-based. Appending `-sessid-{id}` to the username breaks authentication. `refreshProxySession()` handles this automatically.
- DataImpulse returns `407 TRAFFIC_EXHAUSTED` when the account balance runs out. Without the `proxyExhausted` flag, this caused `ERR_PROXY_AUTH_UNSUPPORTED` on every fetch AND browser request, killing the entire scraper. Now: fetch paths detect 407 and skip immediately; browser paths on Mac bypass the proxy entirely (direct residential IP).
- Playwright requires proxy credentials as separate `{ server, username, password }` fields — passing the full URL (like `HttpsProxyAgent` accepts) causes `ERR_INVALID_AUTH_CREDENTIALS`.
- `got-scraping` response API differs from `node-fetch`: `statusCode` (not `status`), `body` as string (not stream), `proxyUrl` option (not `agent`), `timeout: { request: ms }` (not `AbortSignal`). The `scraperFetch()` wrapper normalizes this to a node-fetch-compatible interface.
- `got-scraping` `useHeaderGenerator` is set to `false` in `scraperFetch()` — we supply our own Chrome-accurate `FETCH_HEADERS` rather than relying on got-scraping's auto-generated headers.
- **Kroger `fulfillment.inStore: true` is a planogram signal, NOT real-time stock.** It correlates with the store having a designated shelf slot for the product, not whether anything is currently on that shelf. The B+C tier classifier (`classifyKrogerFind`) discriminates: confirmed = facings ≥ 3 OR price `effectiveDate` within 14 days; lead = single-facing slot with stale price data. Lead alerts get yellow Discord embed + "call ahead" guidance.
- **Kroger product URLs require slug + productId**: `/p/0008800402784` (productId-only) returns 404 for some bottles. Always use the full slug from the API's `productPageURI` field via `buildKrogerProductUrl(product, productId)`. URL-only finds in old `state.json` were retroactively repaired.
- **Fry's website is behind Akamai** — direct fetches to product pages return Access Denied. The Atlas API (`/atlas/v1/product/v2/products`) requires a complex `x-laf-object` header containing `facilityId` + `assortmentKeys` (per-store, not in our store data) plus a successful POST to `/atlas/v1/modality/preferences` to set the user's preferred store. Setting modality preferences gets `429 cpr_chlge` then escalates to `403` for the remainder of the session — Akamai's "tarpit" defense. The HTML verify path (`KROGER_HTML_VERIFY=true`) is implemented but disabled by default because of this.
- **Albertsons uses Incapsula, not Akamai** — `incap_ses_*` and `visid_incap_*` cookie names instead of Akamai's `_abck`. Same browser-first pattern, but cookie cache and challenge handling differ. Same Azure APIM API as Safeway (same parent company), so `SAFEWAY_API_KEY` works on `albertsons.com` once past Incapsula.
- **Safeway also moved to Incapsula in 2026** — historically Akamai, but a browser session against `safeway.com` now earns `visid_incap_*`/`incap_ses_*`/`nlbi_*` cookies and zero Akamai. The browser-mediated API path still works because it inherits whatever WAF cookies the page sets, but the fetch-first path (which expected cached `_abck` from prior browser scrapes) silently no-ops. Treat Safeway and Albertsons as the same WAF playbook.
- **Costco's 2026 redesign dropped `[data-testid="search-results"]`** as a content anchor. The browser scraper's "is this a degraded page or a genuine empty result?" check now uses `[data-testid^="LinkList"]`, `[data-testid="MarkdownRenderer"]`, and `main h1/h2` as fallbacks — those persist on the new design. Without this, every successful query with no products got marked as "blocked" and triggered the 30-min adaptive cooldown.
- **Granular failure-mode tracking** — `trackHealth(key, outcome, { reason })` accepts a reason in `{waf, proxy, soft_block, contract_drift, timeout, network}`. Counted in `scraperHealth[key].reasons`. Surfaces root cause in metrics so future audits don't require manual probing — distinguishes "Akamai blocked us" (waf) from "DataImpulse exhausted credits" (proxy) from "page contract changed" (contract_drift) from "no products but page healthy" (soft_block).
- **Per-retailer canary configuration** — `CANARY_BY_RETAILER` config map + `isCanaryFor(retailerKey, bottleName)` helper. Default canary is Buffalo Trace (`canary: true` in TARGET_BOTTLES), but retailers that don't reliably stock BT (Walgreens, Costco) can override. Empty array means "don't expect a canary here" — degradation alerts skip those retailers to avoid false alarms.
- **Health-degradation Discord alert** — `detectHealthDegradation(recentMetrics)` flags retailers with 4+ consecutive missed-canary scans + `maybeSendHealthDegradationAlert()` fires a one-time orange Discord embed with the dominant `reason` from the failure-mode counters. Once-per-degradation guard via `healthDegradedRetailers` Set; clears when retailer recovers (canary found again). Catches WAF rotations within ~2 hours instead of leaving them invisible until a full audit.
- **Smart re-alert by rarity** — `rerunCadenceFor(bottle)` returns 1 (re-alert every scan) for `ULTRA_RARE_BOTTLES` (Pappy 15/20/23, Stagg, KoK, Old Forester Birthday, etc.) and the default `REALERT_EVERY_N_SCANS` for common bottles. Means seeing Pappy 23 still-in-stock pings every scan; seeing Buffalo Trace doesn't.
- **Bottle interest tiers** — `BOTTLE_INTEREST_TIERS = { obsess: [...], track: [...], ignore: [...] }`. Obsess = ULTRA_RARE_BOTTLES (every-scan re-alert). Track = default cadence. Ignore = silently tracked in state.json but no Discord alerts (useful for "found but not buying" cases). `IGNORED_BOTTLES` Set is checked in recordResult to filter alertableFinds.
- **Watchlist as JSON file** — `WATCH_LIST_DEFAULTS` (in-code defaults) merged with `watchlist.json` (gitignored, edit-without-deploy) on every poll. Live-reload — no daemon restart needed to add a tip. `loadWatchList()` runs at start of `poll()`.
- **Price history per bottle** — `state.bottles[name].priceHistory = [{ts, price}, ...]` capped at `PRICE_HISTORY_LIMIT = 30` entries. Append-only; only adds when price differs from last entry (no spam logging the same value). Used by `scripts/seasonal-patterns.js` and price-change detection.
- **Auto-confirmation 24h** — bottles continuously seen for 24+ hours get a `confirmedAt` timestamp in state.json, surfacing as 🔵 in `formatBottleLine`. Distinguishes durable real stock from short-lived planogram artifacts. Sticky once set; cleared only when bottle goes OOS.
- **Cross-source confirmation tier** — `state._redditMentions` records Reddit-mentioned bottle timestamps during `scrapeRedditIntel`. In `recordResult`, any newFind with a Reddit mention in the last 4 hours gets `crossSourceConfirmed: true` → 🔥 "DOUBLE-CONFIRMED" badge in `formatBottleLine`. Highest-confidence find: human reporting + scraper agree.
- **Alert dedup across stores** — `urgentlyAlertedThisScan` Set tracks bottle names already pinged with @here in the current poll. Subsequent same-bottle finds at other stores downgrade to quiet alerts with "🎯 ALSO AT —" titles. Reduces Discord noise when one bottle drops at multiple Marketplace stores simultaneously.
- **Bandwidth filter on browser scrapers** — `applyBandwidthFilter(page)` blocks `image`/`font`/`media` resource types and 17 ad/analytics domains via `page.route()`. Cuts proxy bandwidth ~70% and scan time 20-40%. Critical: stylesheets are NOT blocked (some sites gate JS execution on CSS load).
- **CityHive multi-tenant platform — category iteration, not search** — Two stores currently active (ExtraMile, Chandler Liquors); a third (Liquor Express Tempe) is configured but disabled because its homepage doesn't render categories the same way. All three share `scrapeCityHiveStore` via `makeCityHiveScraper(retailerKey)`. **Critical learning** (from a session of trial and error): CityHive's `browse_categories/render.json` endpoint takes a `product_filtered_group_id` (a hardcoded category ID per merchant) — NOT a search query. The SPA's "search" feature is purely client-side filtering over multiple category renders. Each retailer in `CITYHIVE_RETAILERS` declares a `categories: [{id, title}]` array of bourbon-bearing categories discovered by browser-probing the homepage. Empty `categories[]` = silent skip without consuming health budget. `matchCityHiveProducts(products, retailerKey, domain)` checks `merchants[0].offer_types` (non-empty array = in-stock); falls back to `is_buyable: true`. Adding a new CityHive store: probe homepage in DevTools, capture `browse_categories` POST bodies, note category IDs that return bourbon products, populate `CITYHIVE_RETAILERS[key].categories`. Some stores (like Liquor Express Tempe) use a different layout with no auto-rendered categories — these need browser-mediated scraping which we haven't built. **Honest expectation**: small independent stores list their general bourbon catalog but allocated drops (Pappy/BTAC) typically aren't in online inventory — store owners reserve them for walk-ins. Scraper picks up canary + common bourbons; allocated finds at these stores will be opportunistic.
- **Walmart rejection-reason diagnostic** — When `matchWalmartNextData` finds a name match but rejects the item (sellerName, availability, fulfillment), logs the specific filter that fired. Walmart has produced 0 non-canary finds across 290+ scans; this surfaces whether matches exist but get filtered (vs. no allocated stock at our stores).
- **Reddit → watch list bridge** — `inferRetailerFromText(text, keywords)` extracts the most-mentioned retailer from a Reddit post body, used in `scrapeRedditIntel` to auto-create a `_watchList` entry for each match. Records the intelligence; doesn't auto-create proactive scans (the Reddit alert IS the user-facing signal). Pre-marked as already-notified so the existing `processWatchList` doesn't fire twice.
- **Two-tier summary embed** — `buildSummaryEmbed` accepts `totalConfirmed` + `totalLeads` and shows them split when present. The `poll()` aggregator splits `changes.newFinds` by `confidence` to compute the tier counts. Lets a quick glance at the summary differentiate "5 confirmed (drive over)" from "5 leads (call first)".

## Modularization plan

`scraper.js` is ~6200 lines. Splitting it lowers regression risk and reduces context cost when iterating on a single retailer. Strategy: **shared deps first, then per-retailer slices**, each phase a self-contained PR with green tests.

### Phase 1 — done ✅

`lib/bottles.js` extracts pure data + pure functions: `SEARCH_QUERIES`, `TARGET_BOTTLES`, `CANARY_NAMES`, `CANARY_BY_RETAILER`, `isCanaryFor`, `BOTTLE_INTEREST_TIERS`, `ULTRA_RARE_BOTTLES`, `IGNORED_BOTTLES`, `bottleInterestTier`. `scraper.js` imports + re-exports for backward-compat. Zero test changes; 907/907 green.

### Phase 2 — shared HTTP + matching layer — done ✅

Three more pure-ish modules extracted, no behavior changes:

1. **`lib/blocked.js`** — `isBlockedPage`, `isFetchBlocked`, `isCostcoBlocked` (alias), `FETCH_BLOCKED_PATTERNS`, `solveHumanChallenge`. Body-text heuristics — no env, no state. Inlines its own jittered `sleep` to stay dependency-free.
2. **`lib/match.js`** — `normalizeText`, `EXCLUDE_TERMS`, `matchesBottle`, `parsePrice`, `dedupFound`, `parseSize`, `truncateTitle`, `DISCORD_TITLE_LIMIT`, `filterMiniatures`, `MIN_BOTTLE_PRICE`, `MAX_BOTTLE_PRICE`. All pure functions or pure constants.
3. **`lib/http.js`** — `IS_MAC`, `CHROME_VERSION` (auto-detected from system Chrome at import time, <50ms execFileSync), `CHROME_PATH`, `getGreasedBrand`, `FETCH_HEADERS`, `fetchRetry`. **Decision**: `scraperFetch` / `scraperFetchRetry` stayed in `scraper.js` rather than moving here because they're tightly coupled to the proxy-exhaustion state machine (`primaryProxyExhausted`, `failoverToBackupProxy`, `trackBandwidth`). Pulling that into a shared module would balloon the regression surface — future per-retailer modules import `FETCH_HEADERS` from here and call `scraperFetch` from scraper.js, which keeps proxy state in one place.

After Phase 2, per-retailer modules in Phase 3 only need to import `lib/bottles.js`, `lib/match.js`, `lib/http.js`, `lib/blocked.js` + a small set of orchestration helpers (`trackHealth`, `acquireRetailerLock`, `getCachedCookies`, `cacheRetailerCookies`, plus `scraperFetch` itself).

**Discovered + fixed during Phase 2**: the bottom test-export block in scraper.js (`export { ... }` at end of file) duplicated names already re-exported at the top via `export { ... } from "./lib/X.js"`. Vitest tolerates duplicate exports (esbuild dedups silently); raw Node ESM rejects with `Duplicate export of 'X'`. Phase 1 had quietly shipped with this bug — daemon hadn't restarted since the push. The Phase 2 commit fixes both phases by removing all moved names from the bottom export block.

### Phase 3 — per-retailer modules (planned)

One PR per retailer, in order of independence (least intertwined first):

1. `lib/retailers/walgreens.js` — browser-only, scrapeOnce, no fetch path. Smallest surface area. (~150 lines)
2. `lib/retailers/samsclub.js` — fetch-first per-product, scrapeOnce. Uses `SAMSCLUB_PRODUCTS` map. (~300 lines)
3. `lib/retailers/totalwine.js` — fetch-first via `INITIAL_STATE`, browser fallback. (~400 lines)
4. `lib/retailers/walmart.js` — fetch-first via `__NEXT_DATA__`, browser fallback. (~400 lines)
5. `lib/retailers/costco.js` — fetch-first + browser known-URL fallback. Most complex, save for last. (~600 lines)
6. `lib/retailers/kroger.js` — pure API, but lots of B+C tier classification logic. (~500 lines)
7. `lib/retailers/safeway.js` + `lib/retailers/albertsons.js` — same Incapsula playbook, can share helpers. (~350 lines each)
8. `lib/retailers/cityhive.js` — already largely factored via `makeCityHiveScraper`. Lift to its own file. (~200 lines)

Each retailer file exports `scrape{Name}Store` (or `scrape{Name}Once` for scrapeOnce retailers) plus any retailer-specific match helpers. `scraper.js` imports them and registers in `RETAILERS`.

### Phase 4 — orchestration cleanup (planned)

Once retailers are out, what remains in `scraper.js` is the poll loop, Discord embedding, state management, and watch list / Reddit intel. Consider splitting:

- `lib/state.js` — `loadState`, `saveState`, `pruneState`, `updateStoreState`, `computeChanges`, `loadKnownProducts`.
- `lib/discord.js` — `postDiscordWebhook`, `buildStoreEmbeds`, `buildSummaryEmbed`, `sendUrgentAlert`, `sendDiscordAlert`, `formatBottleLine`, `formatStoreInfo`.
- `lib/intel.js` — `scrapeRedditIntel`, `loadWatchList`, `processWatchList`.
- `scraper.js` becomes a slim entry point: `poll()`, `main()`, schedule logic, env validation.

**Regression-minimization rules for every phase:**
- Each phase imports + re-exports through `scraper.js` so test imports keep working.
- No behavioral changes during the move — extraction-only commits.
- Run full test suite after every phase. 907/907 must stay green.
- One module per commit. Easy bisect if something breaks.

## Auxiliary scripts

- **`scripts/analyze-drops.js [--days=14] [--retailer=kroger]`** — Reads `metrics.jsonl` and surfaces day-of-week × hour patterns where allocated bottles actually appear in stock. Top-15 drop-time slots ranked by hit rate; per-retailer canary health timeline; first-find timestamps per bottle. Used to validate boost-schedule tuning.
- **`scripts/audit-state.js [--fix] [--days=30]`** — Audits `state.json` for stale entries, missing prices, broken URLs, orphaned stores, expired Reddit-spawned watch list entries. Read-only by default; `--fix` applies automatic repairs (currently: prune stale watch list entries).
- **`scripts/seasonal-patterns.js [--bottle=X] [--days=365]`** — Bottle × month heatmap from `metrics.jsonl`. Surfaces drop windows for specific bottles (BTAC in Oct, KoK quarterly, etc.). Top-3 peak months per bottle + boost-window predictions for current/next month. Useful for tuning advance hunting strategy.
- **`scripts/find.js [query] [--retailer=X] [--hot] [--confirmed] [--price-max=N]`** — Multi-retailer unified CLI search across `state.json`. Lists current in-stock bottles grouped by name, shows tier (FOUND/CONFIRMED/LEAD), per-store details, age since last-seen. Replaces grepping JSON manually.
- **`scripts/map.js [--output=path]`** — Renders current finds as a Leaflet map (self-contained HTML, no server). Tier-colored markers (red=double-confirmed, blue=24h-confirmed, green=found, yellow=lead). Click a pin → popup with bottles + prices + product page links. Geographic view of "where's allocated bourbon right now?"
- **300s browser timeout is the floor**, not a margin. Pre-warm alone takes 130-180s on modern WAFs because `networkidle` never settles (sensors keep beaconing). 180s = guaranteed failure (queries get 0 budget after pre-warm). 300s = ~120-170s of real query time. Bumping further would help marginally; the right fix is replacing `networkidle` with a smarter wait (open work).
