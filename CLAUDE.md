# Bourbon Scout

Allocated bourbon inventory scraper with Discord webhook alerts. Monitors 7 retailers for 39 bottles (38 rare/allocated + 1 canary health check) near a zip code, sends per-store Discord alerts with @everyone pings when bottles are found, plus a quiet summary with per-scraper health metrics after each scan.

## Architecture

- **`scraper.js`** — Main entry point. Contains all scrapers, Discord alerting, and poll orchestration. Runs a setTimeout-based poll loop with ±3 min jitter (or single-run via `RUN_ONCE=true` for CI). Scans all stores concurrently with a limit of 8.
- **`browser-profiles/`** — Per-retailer persistent Chrome profile directories (auto-created). Contains HTTP cache, service workers, IndexedDB, visited history. Gitignored.
- **`lib/geo.js`** — Zip-to-coordinates (`zipToCoords`) via zippopotam.us API and `haversine` distance calculation. No dependencies.
- **`lib/discover-stores.js`** — Auto-discovers nearby stores for all retailers given a zip code and radius. Caches results to `stores.json` with a 7-day TTL. Uses Playwright for browser-based store locators and Kroger REST API for Kroger.
- **`lib/fallback-stores.js`** — Static hardcoded store data for the 85283 area. Used when browser-based store locators fail (e.g., CI datacenter IPs get blocked).
- **`state.json`** — Per-store stock state, nested as `{ "retailer": { "storeId": { "bottles": { "name": { url, price, sku, firstSeen, lastSeen, scanCount } }, "lastScanned" } } }`. Auto-generated, safe to delete for a fresh start.
- **`stores.json`** — Cached store discovery results. Auto-generated, safe to delete to force re-discovery.
- **`.github/workflows/bourbon-scout.yml`** — GitHub Actions cron job. Runs every 15 min (offset to `:03/:18/:33/:48`). Runs on self-hosted Mac runner (`bourbon-scout-mac`, residential IP). Caches state files between runs. Uses `RUN_ONCE=true`. 25-minute timeout.
- **`debug/`** — Screenshots, HTML dumps, and CSS selector lists captured from retailer store locator pages. Used for fixing selectors when sites change. Generated by `debug-locators.js`.

## Key Patterns

- **ESM throughout** — `"type": "module"` in package.json, all files use `import`/`export`
- **rebrowser-patches + Stealth + System Chrome** — `rebrowser-playwright-core` (patched Playwright fork) wrapped with `playwright-extra`'s `addExtra()` and `puppeteer-extra-plugin-stealth`. rebrowser-patches rewrites CDP commands (Runtime.Enable, sourceURL, utility world naming) to evade Akamai/PerimeterX automation detection at the protocol level. Stealth plugin handles higher-level browser fingerprint spoofing (navigator.webdriver, chrome.runtime, etc.). On Mac, launches system Chrome (`/Applications/Google Chrome.app`) via `executablePath` for authentic TLS fingerprint. Falls back to bundled Chromium on other platforms. Launches with `--disable-blink-features=AutomationControlled` and `ignoreDefaultArgs: ['--enable-automation']`. Browser contexts get platform-aware Chrome 145 UA + `Sec-CH-UA` Client Hints matching `FETCH_HEADERS`.
- **Platform-aware User-Agent** — `IS_MAC` detection at module level. macOS Chrome UA + `Sec-CH-UA-Platform: "macOS"` on Mac, Windows Chrome on Linux/CI. Prevents UA-vs-TLS fingerprint mismatch that bot detectors flag.
- **Lazy browser launch** — Browser is only launched when a scraper actually needs it (via `newPage()`). When all scrapers succeed via fetch-first, no browser is started.
- **Atomic state writes** — `saveState()`, `saveBrowserState()`, `saveCache()` (discover-stores) all write to `.tmp` then `rename()` (atomic on POSIX). Prevents corruption on crash mid-write.
- **Browser cookie persistence** — `loadBrowserState()`/`saveBrowserState()` persist Playwright `storageState` to `browser-state.json`. In-memory cache (`browserStateCache`) avoids re-reading from disk on every `newPage()`. Cookies accumulate across polls, reducing bot detection signals.
- **Parallel query execution** — API scrapers (Kroger, Safeway) run all 13 queries via `Promise.all` with 50ms staggered starts. Fetch scrapers (Costco, TotalWine, Walmart) use batched parallelism via `runWithConcurrency(queryTasks, 4)`.
- **Kroger pagination** — Fetches page 2 (`filter.start=50`) when first page returns exactly 50 results, catching allocated bottles that spill past the first page. Same pattern used by Safeway.
- **Kroger null-inventory guard** — Requires `stockLevel != null` in addition to checking for `TEMPORARILY_OUT_OF_STOCK`. Prevents false positives when Kroger API returns products with null/missing inventory data.
- **Concurrent scanning** — All stores run concurrently via `runWithConcurrency(tasks, 8)`. Browser and API scrapers are unified in the same task queue.
- **Structured data extraction over CSS selectors** — Scrapers prefer embedded JSON (Walmart `__NEXT_DATA__`, Total Wine `window.INITIAL_STATE`) over fragile CSS selectors. Costco uses stable MUI `data-testid` attributes. CSS selector fallbacks exist but are last resort.
- **Residential proxy support** — Optional `PROXY_URL` env var routes all scraper traffic (browser + fetch) through a proxy. Supports HTTP/HTTPS and SOCKS5/SOCKS4 protocols — `createProxyAgent()` auto-detects based on URL scheme. Playwright browser proxy requires URL parsing into `{ server, username, password }` (can't pass the full URL). Discord webhooks skip the proxy. With proxy set, Walmart fetch-first works on CI too.
- **Sticky proxy sessions** — `refreshProxySession()` called at the start of each `poll()`. For DataImpulse, switches to a random sticky port (10000-20000) so all requests in a scan share one residential IP for ~30 min. Prevents IP rotation from breaking cookie chains (e.g., Costco homepage pre-warm cookies tied to one IP, search requests from another).
- **Per-retailer IP isolation** — Each retailer gets its own sticky proxy port (= its own residential IP) via `retailerProxyAgents`. Prevents cross-retailer IP flagging cascades — if Costco's IP gets burned, Walmart's is unaffected. `getRetailerProxy(key)` returns the per-retailer fetch agent; `getRetailerBrowserProxy(key)` returns Playwright-compatible `{ server, username, password }`.
- **Persistent browser profiles** — `launchRetailerBrowser(retailerKey)` uses `chromium.launchPersistentContext(profileDir)` to maintain full Chrome profiles per retailer in `browser-profiles/{key}/`. HTTP cache, service workers, IndexedDB, and visited history persist on disk across polls — the scraper looks like a returning visitor. Within a poll, `retailerBrowserCache` shares the context across stores. Pages are closed after each use; contexts closed at end of poll via `closeRetailerBrowsers()`. Profile data survives context close for next poll.
- **Query rotation across scans** — `getQueriesForScan(SEARCH_QUERIES)` alternates which queries run based on `scanCounter % 2`. Each scan fires ~8 queries instead of 14 (half the non-canary queries + always the canary). All 39 bottles are covered within 2 consecutive scans (30 min at 15-min intervals). Reduces per-session "hammering" that bot detection ML flags.
- **Query order randomization** — `shuffle(getQueriesForScan(SEARCH_QUERIES))` via Fisher-Yates before every scraper loop to avoid predictable access patterns that anti-bot ML models flag.
- **Chrome TLS fingerprint impersonation (got-scraping)** — All scraper fetch paths use `got-scraping` from Apify instead of `node-fetch`. `got-scraping` impersonates Chrome's TLS handshake (JA3/JA4 fingerprint, cipher suites, extensions order), making HTTP requests indistinguishable from real Chrome at the network layer. `scraperFetch()` wraps `gotScraping()` with a node-fetch-compatible response interface (`{ ok, status, headers, text(), json() }`). `scraperFetchRetry()` adds one-retry-on-error. API scrapers (Kroger, Safeway) and Discord webhooks still use raw `node-fetch` (no bot detection). Proxy passed as `proxyUrl` string option (not `agent` object).
- **Fetch-first with browser fallback** — All 5 browser-based scrapers try `got-scraping` fetch first (with Chrome TLS fingerprint) before spinning up Playwright. Walmart parses `__NEXT_DATA__`, Sam's Club parses `__NEXT_DATA__` per-product, Costco parses `data-testid` tiles via cheerio, Total Wine extracts `INITIAL_STATE` JSON. Falls back to dedicated browser instances if blocked.
- **Per-product-URL scraping** — Sam's Club search excludes spirits, so `SAMSCLUB_PRODUCTS` maps bottle names to product IDs. Each product page is fetched individually at `/ip/{productId}` instead of using SEARCH_QUERIES. Only bottles with known Sam's Club pages are checked.
- **Fetch retry** — `fetchRetry()` (for `node-fetch` API paths) and `scraperFetchRetry()` (for `got-scraping` scraper paths) retry once on network errors with 1s backoff and `console.warn` on first failure.
- **Brace-counting JSON extraction** — Total Wine INITIAL_STATE and Walmart `__NEXT_DATA__` are both extracted from HTML using a character-by-character brace counter that tracks string context and escape characters, safely handling `</script>` inside JSON string values.
- **Scrape-once pattern** — Costco search has no store filter, so results are identical across warehouses. Walgreens uses `USER_LOC` cookie with zip coordinates — nearby stores share results. Sam's Club product pages show availability for the nearest club. All three scraped once and broadcast to all stores.
- **Broad search queries** — 14 queries cover all 39 bottles (38 allocated + 1 canary). API scrapers (Kroger, Safeway) use these instead of per-bottle queries (~60% fewer API calls).
- **Canary bottle (Buffalo Trace)** — Always-in-stock bottle used as a scraper health check. Found canary = scraper is working. Missing canary = scraper may be broken. Canary results are filtered out before alerts/state — only shown in the health section of the summary embed.
- **Per-scraper health tracking** — `scraperHealth` module-level object tracks queries succeeded/failed/blocked per retailer per poll. `trackHealth(retailerKey, outcome)` called in every scraper's query loop. Reset each poll. Displayed as inline Discord embed fields (3 per row).
- **Shared Kroger OAuth token** — Fetched once per poll via singleton promise, shared across all Kroger stores. Clears cached token on 401 for automatic re-auth. Uses promo price when available (falls back to regular).
- **Per-scraper browser timeout with health tracking** — `withTimeout(promise, 180000, null)` wraps all browser scraper calls (Costco, Total Wine, Walmart, Walgreens, Sam's Club). On timeout (3 min), wrappers log a warning, call `trackHealth(key, "blocked")`, and return `[]`. This ensures adaptive retailer skipping engages for completely hung browsers (Akamai challenge pages that never resolve), rather than silently treating them as empty results.
- **Homepage pre-warm with humanization** — All browser scrapers (Costco, Total Wine, Walmart, Walgreens, Sam's Club) visit the retailer homepage before searching. Uses `networkidle` wait + 3-5s randomized dwell time to let anti-bot sensor scripts (Akamai `_abck`, PerimeterX `_px*`) fully execute and phone home. After dwell, `humanizePage(page)` scrolls down via mouse wheel events, hovers over a random link, then scrolls back up using wheel events (not `scrollTo` — that's a JS teleport detectable by anti-bot sensors). Costco fetch path also pre-warms, collecting `Set-Cookie` headers for subsequent requests.
- **Category navigation flow** — After homepage pre-warm, browser scrapers navigate to a spirits/liquor category page (`navigateCategory(page, key)`) before searching. This simulates a real user flow: homepage → browse category → search. `CATEGORY_URLS` maps retailer keys to known category page URLs (e.g., Costco `/liquor.html`, Total Wine `/spirits/c/spirits`). Non-critical — failures are caught and searching continues.
- **Staggered retailer start times** — In `poll()`, each retailer's first task sleeps 10-30s (randomized) before starting, so all 7 retailers don't hit the internet simultaneously. First retailer starts immediately; subsequent ones stagger.
- **Direct product URL checks** — `loadKnownProducts(state)` extracts previously found bottle URLs from `state.json` at the start of each poll. `checkWalmartKnownUrls(store)` checks known Walmart product pages directly before running the full search — less suspicious than searching and catches bottles search might miss.
- **Variable scan intervals** — Uses `setTimeout` chain with ±3 min jitter instead of fixed cron. `parsePollIntervalMs()` converts the cron expression to a base interval. Each cycle fires at a different interval (e.g., 12min, 17min, 14min...) — indistinguishable from a human checking occasionally. `node-cron` dependency removed.
- **Adaptive retailer skipping** — `shouldSkipRetailer(key)` / `recordRetailerOutcome(key, success)` track consecutive failures per retailer. After 3 consecutive polls with <25% query success rate, the retailer is skipped for 30 min to let IP reputation recover. Cooldown auto-expires.
- **Human-like inter-query pacing** — Browser scrapers wait 2-4s (randomized) between search queries. All fetch scrapers (Costco, Total Wine, Walmart, Sam's Club) wait 1-2s between queries. Real humans take 5-15s between searches, so this interval is still faster but much less suspicious than sub-second bursts.
- **Early-abort health tracking** — When fetch scrapers hit `failures > 3` and abort remaining queries, each skipped query is tracked as `blocked` via `trackHealth()`. This ensures Discord health summaries show the true query count (all 8, not just the 4 that ran), and adaptive skipping calculations use accurate success rates.
- **Backup proxy provider** — `BACKUP_PROXY_URL` + `BACKUP_PROXY_RETAILERS` env vars let specific retailers use a different proxy provider (e.g., IPRoyal for Costco if DataImpulse IPs are burned at Akamai). Each backup retailer gets its own sticky port on the backup proxy. Unused retailers stay on the primary `PROXY_URL`.
- **Costco fetch resilience** — `scrapeCostcoViaFetch` has layered anti-bot countermeasures: (1) cookie pre-warm from homepage, (2) adaptive concurrency (1 for SOCKS5, 4 for HTTP), (3) inter-query jitter (250-500ms), (4) expanded blocked detection via `isCostcoBlocked()` with 8 Akamai patterns, (5) retry-with-backoff on blocked responses (2-3s pause before retry).
- **Bot detection** — `isBlockedPage()` checks page title AND body text for challenge/CAPTCHA/access-denied/security-check indicators. Browser scrapers skip queries that hit bot detection instead of recording false "empty" results. Costco fetch path uses `isCostcoBlocked()` with additional Akamai-specific patterns (`_ct_challenge`, `Request unsuccessful`, `Incapsula`, `Enable JavaScript`, `verify you are human`).
- **Per-retailer Sec-Fetch-Site** — First query per scraper sends `cross-site` + Google Referer (simulating search click). Subsequent queries send `same-origin` + retailer domain Referer (simulating internal navigation). Uses `.map()` index (`i > 0`) instead of runtime flag to avoid race conditions in concurrent task execution.
- **Chrome-accurate FETCH_HEADERS** — `Accept` header matches real Chrome 145 (includes `image/avif`, `image/webp`, `application/signed-exchange`). `Sec-Fetch-User: ?1` header included. `Accept-Encoding` is `gzip, deflate` only (Brotli removed for compatibility).
- **State pruning** — `pruneState()` removes stale retailer/store entries from `state.json` on each poll, keeping state clean when stores change.
- **Text normalization** — `normalizeText()` converts Unicode curly quotes/apostrophes to ASCII before matching, preventing silent misses when retailers use `\u2019` in product names.
- **Fetch sanity checks** — Walmart fetch path validates `__NEXT_DATA__` contains a real `searchResult` structure before trusting results. Prevents challenge pages with valid JSON but no search data from being treated as "nothing in stock."
- **AbortSignal timeouts** — All fetch calls (Discord 10s, zipToCoords 10s, Kroger/Safeway/Walmart 15s, Kroger store locator 15s) use `AbortSignal.timeout()` to prevent hung requests from blocking concurrency slots.
- **Error isolation** — `runWithConcurrency` wraps tasks in `.catch()` so one failing task never aborts remaining tasks. Costco broadcast loop has per-store try/catch. `saveState`/`closeBrowser`/summary send are all individually error-handled.
- **Store discovery is separate from scraping** — `discover-stores.js` handles finding stores, `scraper.js` handles checking inventory
- **State change tracking** — `computeChanges()` diffs previous and current scan results. `updateStoreState()` persists `firstSeen`, `lastSeen`, `scanCount` per bottle per store. Re-alerts every N scans via `REALERT_EVERY_N_SCANS`.
- **Color-coded Discord embeds** — Green `@everyone` for new finds, orange for OOS losses, blue re-alerts, purple scan summary. Summary includes scanned store list (grouped by retailer) when nothing is found. 429 rate limit retry with `Retry-After` backoff (throws after 3x 429 to prevent silent alert loss). Embed titles truncated at 256 chars, descriptions at 4096 chars.
- **Google Maps links** — Store addresses are hyperlinked to Google Maps universal search URLs.

## Environment Variables

All config is in `.env`:
- `ZIP_CODE` — center point for store discovery (currently 85283 / Tempe, AZ)
- `SEARCH_RADIUS_MILES` — how far to search (default 15)
- `MAX_STORES_PER_RETAILER` — caps per chain (default 5)
- `POLL_INTERVAL` — base scan interval as cron expression (default `*/15 * * * *`). Parsed to minutes for setTimeout-based scheduling with ±3 min jitter.
- `REALERT_EVERY_N_SCANS` — re-alert interval for bottles still in stock (default 4)
- `KROGER_CLIENT_ID`, `KROGER_CLIENT_SECRET` — Kroger API OAuth credentials
- `SAFEWAY_API_KEY` — Safeway product search API key
- `DISCORD_WEBHOOK_URL` — Discord webhook for alerts
- `PROXY_URL` — Residential proxy URL. Supports HTTP (`http://user:pass@gw.dataimpulse.com:823`) and SOCKS5 (`socks5://user:pass@host:1080`). Routes browser and fetch scraper traffic through proxy. Discord webhooks are NOT proxied. DataImpulse sticky sessions use ports 10000-20000 (auto-configured by `refreshProxySession()`).
- `BACKUP_PROXY_URL` — Optional backup proxy URL (e.g., IPRoyal). Used by retailers listed in `BACKUP_PROXY_RETAILERS`.
- `BACKUP_PROXY_RETAILERS` — Comma-separated list of retailer keys to route through `BACKUP_PROXY_URL` instead of `PROXY_URL` (e.g., `costco,totalwine`).
- `RUN_ONCE` — Set to `true` for single-run mode (used by GitHub Actions)

## Debugging Store Locators

When a retailer's store locator stops finding stores (selectors broke):
1. Run `node debug-locators.js` to capture fresh screenshots + HTML + selector lists to `debug/`
2. Compare `debug/{retailer}-selectors.txt` against the selectors in `lib/discover-stores.js`
3. Update the locator function, delete `stores.json`, and re-run

## Gotchas

- Costco's search URL (`/s?keyword=`) returns identical results regardless of warehouse — that's why we scrape once and broadcast.
- Total Wine uses PerimeterX (HUMAN Security) — `got-scraping` with Chrome TLS fingerprint + residential proxy significantly improves fetch success rate. Falls back to browser if blocked.
- Walmart fetch-first gets blocked on CI (datacenter IPs) but works locally. With `PROXY_URL` set, fetch-first works on CI too. Without proxy, fetch path skips on CI and falls back to browser.
- Walmart fetch validates `__NEXT_DATA__` has a `searchResult` key — challenge pages with valid JSON but empty search data correctly fall back to browser instead of reporting "nothing found."
- Store locator CSS selectors in `discover-stores.js` break when retailer sites update their DOM. Kroger's API locator is the most stable.
- Safeway's store locator is at `local.safeway.com/search.html` — NOT `safeway.com/stores/search` (that 404s).
- Deleting `state.json` is safe but will send a full report for all stores on next poll.
- The Safeway scraper and Kroger scraper silently skip if their API keys aren't set.
- `stores.json` cache invalidates on zip/radius change or after 7 days.
- `lib/fallback-stores.js` provides static store data when browser-based locators fail on CI.
- Walgreens has no embedded JSON (`__NEXT_DATA__`, `INITIAL_STATE`) — server-rendered HTML only. Akamai blocks direct fetch (403), so it's browser-only with no fetch-first path. Store context set via `USER_LOC` cookie (base64 JSON: `{"la":"<lat>","lo":"<lng>","uz":"<zip>"}`).
- Sam's Club search (`samsclub.com/search?q=`) does NOT return spirits — product pages exist at `/ip/{slug}/{id}` but are excluded from search results. The scraper uses `SAMSCLUB_PRODUCTS` (a static map of bottle→productId) instead of `SEARCH_QUERIES`. Not all 39 bottles have Sam's Club pages — only 8 are currently mapped.
- NordVPN SOCKS5 proxy (`phoenix.us.socks.nordhold.net:1080`) doesn't work for scraping — it caps concurrent connections, causing `ConnectionRefused`/`ERR_SOCKS_CONNECTION_FAILED` under load.
- `PROXY_URL` supports both HTTP and SOCKS5 via `createProxyAgent()` auto-detection, but SOCKS5 proxies with connection limits (like NordVPN) will fail under concurrent load. Use a residential HTTP proxy (DataImpulse at $1/GB recommended).
- DataImpulse sticky sessions are **port-based** (10000-20000), NOT username-suffix-based. Appending `-sessid-{id}` to the username breaks authentication. `refreshProxySession()` handles this automatically.
- Playwright requires proxy credentials as separate `{ server, username, password }` fields — passing the full URL (like `HttpsProxyAgent` accepts) causes `ERR_INVALID_AUTH_CREDENTIALS`.
- `got-scraping` response API differs from `node-fetch`: `statusCode` (not `status`), `body` as string (not stream), `proxyUrl` option (not `agent`), `timeout: { request: ms }` (not `AbortSignal`). The `scraperFetch()` wrapper normalizes this to a node-fetch-compatible interface.
- `got-scraping` `useHeaderGenerator` is set to `false` in `scraperFetch()` — we supply our own Chrome 145-accurate `FETCH_HEADERS` rather than relying on got-scraping's auto-generated headers.
